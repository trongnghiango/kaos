## 2. Root Cause Analysis
1.  **Empty/Stale Graph in Redis**: The Act Phase loads tasks in-memory (`self.tasks`) from the cached `scout_report.json`. However, `TaskQueueEngine._calculate_levels()` initially calculated levels by querying `self.knowledge_graph.calculate_levels()` (RedisGraph).
2.  **No Pre-Sync**: The tasks had not yet been executed, so they were never upserted into Redis (upserts only occurred inside `_execute_single_task` during active task running).
3.  **Partial Graph Output**: Because Redis only contained the single task `FIX_001` (from previous incomplete attempts), `calculate_levels()` in RedisGraph returned only `FIX_001`. Since the returned list of levels wasn't empty, the engine bypassed the in-memory fallback, resulting in the truncation of the other 19 tasks.
4.  **Typo in Fallback Sort**: Even if it fell back to in-memory, a Vietnamese translation typo `优质_queue.append(neighbor)` instead of `next_queue.append(neighbor)` caused syntax issues in cycle-breaking scenarios.
5.  **CLI Report Path Discrepancy**: The CLI tried loading the cached `scout_report.json` from the session-specific `TMP_DIR` (which changes every execution), rather than the persistent `KAOS_WORK_DIR`. Additionally, deserializing list-of-dicts conflict points directly to `ScoutReport` dataclass fields threw instantiation errors.

## 3. Implemented Fixes
*   **File `src/kaos/engine/task_queue_engine.py`**:
    *   **Auto-Sync to RedisGraph**: Added logic to sync (upsert) all current tasks and their respective `DEPENDS_ON` relations into `self.knowledge_graph` before calling `calculate_levels()`.
    *   **Completeness/Consistency Validation**: Added a safety check comparing the task IDs returned by RedisGraph against the loaded tasks in memory. If they do not match, the engine logs a warning and falls back to the in-memory Python sort.
    *   **Typo Correction**: Replaced the translation typo `优质_queue` with `next_queue`.
    *   **Resume State Safety**: Modified status assignment so that task status is set to `PENDING` only if it is not already `SUCCESS`, preventing completed tasks from being re-executed during resumes.
*   **File `src/kaos/interfaces/cli.py`**:
    *   Changed cached report path resolution from dynamic `TMP_DIR` to global `KAOS_WORK_DIR / "scout_report.json"`.
    *   Implemented proper deserialization mapping for `ConflictPoint` instances (with enum instantiation) before parsing `ScoutReport`.
